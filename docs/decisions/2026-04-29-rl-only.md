# 决策：主训练路径保持 RL-only

日期：2026-04-29

## 决策

本项目主训练路径只做强化学习，不重新引入 SFT。

## 背景

项目目标是训练 OJ-like code agent，让模型在“公开测试调试 + 正式提交评测”的交互环境中学习写码、调试和提交。当前核心价值来自环境交互、tool feedback、submission reward 和多轮 repair，而不是从旧函数补全样本上做监督微调。

仓库早期存在 SFT、MBPP/HumanEval 和函数 benchmark 方向的实验资产，但这些已经退出当前主线，放在 `docs/legacy/` 仅作历史参考。

## 被拒绝的方案

- **恢复 SFT 作为主训练阶段**：会把优化目标拉回 imitation learning，容易重新围绕旧 benchmark 和函数补全格式设计数据。
- **先 SFT 再 RL 作为默认路线**：短期可能降低冷启动难度，但会扩大数据和协议维护面，模糊当前 OJ-like 环境主线。
- **保留旧 MBPP/HumanEval 训练链路作为并行主线**：会让 reward、prompt、tool 和 evaluation 继续分叉。

## 后果

- 新增训练能力时，优先接入 RL rollout、tool、reward、dataset export 和 eval 链路。
- 不为主线新增 SFT 数据生成、SFT trainer 配置或函数补全协议。
- 历史 SFT 资产可以保留在 `docs/legacy/`，但不是当前规范来源。
- 如果未来重新考虑 SFT，需要先写新的 decision，说明它和 OJ-like RL 主线的边界。
