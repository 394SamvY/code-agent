# debug 文档规范

## 命名规则

`YYYY-MM-DD-<slug>.md`

- 日期前缀保证时间排序
- slug 用英文短横线，概括调试主题
- 示例：`2026-05-01-thinking-budget-detour.md`

## 文档模板

    # <一句话标题>

    日期：YYYY-MM-DD

    ## 背景

    - 什么触发了这次调试？
    - 现象是什么？（日志片段、指标、错误信息）
    - 涉及的环境（模型、GPU、关键配置）

    ## 尝试

    每个 phase 独立小节，记录完整上下文：

    ### Phase N：<方案名>（结果）

    - **动机**：为什么尝试这个方向
    - **改动**：改了哪些代码/配置（文件路径 + 关键变更）
    - **结果**：跑出来的数据（指标变化、错误信息）
    - **结论**：为什么放弃 / 为什么继续

    ## 根因

    真正的原因是什么？（如果走到了根因）

    ## 解决

    - 最终采用的方案是什么？
    - 如果这条路是弯路被废弃了，说明正确的方向在哪、为什么
    - 如果调试未完成，说明卡在哪里、下一步需要什么

    ## 教训

    下次遇到类似问题应该怎么处理？哪些信号可以提前识别？
